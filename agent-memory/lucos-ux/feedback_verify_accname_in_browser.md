---
name: verify_accname_in_browser
description: Adding an aria-label to a child element can silently override a parent link's title-attribute fallback — verify accessible-name computation in a real browser, not just by reading the accname spec or the markup
metadata:
  type: feedback
---

When a link currently has no "name from content" (an empty/unlabelled child, e.g. a placeholder `<div>`) and relies on its own `title` attribute as the accessible name, adding an `aria-label` to that child gives the child its own computed name — which then feeds into the link's "name from content" step and **wins over `title`** per the accname algorithm. The `title` attribute becomes silently unreachable to assistive technology (it may still show as a mouse-hover tooltip, but that's a different, sighted-only channel).

**Why this matters:** on lucos_photos#476, the settled/reviewed spec called for adding a flat `aria-label="No profile picture"` to a placeholder inside a link that today gets its accessible name from `title="{person name/id}"` (homepage.html's people grid). Implementing that literally would have made every nameless/pictureless person's link announce as the identical string "No profile picture" — a real regression, and specifically worse for the population the issue was about (people distinguishable only by their picture). This was caught only by testing with Playwright's `page.locator(...).ariaSnapshot()` before and after the change on the real rendered markup — not by re-reading the ticket's own per-template analysis, which had asserted "zero announcements today" without verifying that claim against the browser's actual accname computation (the `title` fallback was providing a real, if fragile, announcement all along).

**How to apply:** whenever a design adds `aria-label`/`role="img"` to an element nested inside a link, button, or other name-bearing container that might currently be relying on a `title` attribute (or any other low-priority name source) for its accessible name — don't trust a static per-element reading of "this element has no label today, so adding one only helps." Render the before-and-after markup in a real browser (or Playwright's `ariaSnapshot()` against a running dev server) and diff the computed accessible name of the *containing* interactive element, not just the element being edited. If the new label doesn't include whatever identifying information the old fallback provided, fold it in rather than replacing it outright — see [[project_lucos_photos_profile_picture_states]] for the sibling finding from the same ticket about not trusting an un-traced premise (there: DB write-order; here: accname computation).
