`lucos_agent_coding_sandbox` isn't just Lima/agent-VM provisioning — it also has a `pi-hosts/` directory (`setup-unattended-upgrades.sh`, `setup-journald.sh`, README) that is the authoritative provisioning source for xwing/salvare host-level OS config, applied by `lucos_agent_coding_sandbox#22` (closing #21) on 2026-03-06. Any future issue about Pi host OS-level config (unattended-upgrades, journald, sudoers) belongs in this repo's `pi-hosts/`, not `lucos_firewall`, not a new repo, not `lucos`. Confirmed by diffing the script against live `/etc/apt/apt.conf.d/*` and `/etc/sudoers.d/*` on the hosts, not assumed.

**The "security-only" unattended-upgrades restriction on xwing/salvare (`51lucos-security-upgrades`, `Allowed-Origins: Debian:${CODENAME}-security`) is not actually effective as deployed** — found 2026-07-14 while investigating [[agent-sudo-scope]]'s follow-up, `lucos_agent_coding_sandbox#98`. `/usr/bin/unattended-upgrade`'s `get_allowed_origins()` concatenates `Allowed-Origins` entries with `Origins-Pattern` entries rather than letting one override the other. The stock `/etc/apt/apt.conf.d/50unattended-upgrades` file (still present, untouched, ships with the package) contributes an unrestricted `origin=Debian,codename=${distro_codename},label=Debian` pattern — i.e. all stable Debian packages, not just security ones. So routine Debian-origin packages are already being silently auto-upgraded on xwing and salvare, contrary to the #21/#22 design intent. avalon has no custom file at all and is fully on stock (unrestricted-Debian) behaviour.

**Origin labels confirmed via `Release` file inspection** (`grep '^Origin:\|^Label:' /var/lib/apt/lists/*Release`), not inferred from repo URLs: `deb.debian.org` → `Origin: Debian`; `download.docker.com` → `Origin: Docker`, `Label: Docker CE`; `archive.raspberrypi.com` → `Origin: Raspberry Pi Foundation`. None of `unattended-upgrades`' patterns (stock or custom) match Docker or RPi Foundation origins on any host, so those packages structurally never auto-upgrade — this is why #95's backlog exists.

**How to apply:** don't assume the documented intent of a provisioning script (security-only, in this case) matches its actual runtime effect — apt config keys can combine additively across files in ways that silently widen scope past what any single file states. Verify against the tool's actual source when the stakes are "what auto-applies unattended to production," same discipline as [[feedback_read_before_theorising]].

**Resolution direction, agreed with lucos-security on #98 (2026-07-14), pending lucas42's sign-off — do not implement without it:**
- Split into two follow-up issues: (a) fix the security-only concatenation drift, (b) tiered origin coverage (Docker: blanket-allow everywhere, `live-restore: true` already covers the restart risk on all 3 hosts; RPi Foundation: allow userspace/library packages — this is where `libc6`/`openssl` actually live on Pi hardware — but blacklist `linux-image-*`, `linux-headers-*`, `linux-libc-dev`, `raspi-firmware`, `rpi-eeprom`, `firmware-*` via the existing empty `Package-Blacklist` stanza).
- **Correction (team-lead, 2026-07-14): issue (a) is NOT a no-open-questions bugfix — I got this wrong and security converged on the same wrong framing with me.** "Restore the security-only restriction" implies the restriction once held; by the concatenation finding it never did, since #21/#22 deployed in March. So "fixing" it is *enacting an intent for the first time* — a live behaviour change that would **stop** routine Debian-origin auto-upgrades and **increase** manual patching, cutting against the whole point of #95/#98. Supporting evidence: #95's backlog is entirely Docker-origin and RPi-Foundation-origin, zero routine-Debian — consistent with the drift having quietly kept that class patched. Lesson: two agents independently re-verifying the same *mechanism* claim (concatenation behaviour) doesn't catch a shared blind spot in how that mechanism's fix is *framed* — a technically-correct finding can still be wrongly characterised as low-stakes. Don't call something "no open questions" just because the underlying fact is solid; check whether the fix itself is a behaviour change in disguise.
- **salvare (Pi 4) has `rpi-eeprom-update.service` enabled with a pending `rpi-eeprom` bump** — auto-reflashes the physical bootloader EEPROM at next boot, no confirmation step. A bad flash is a physical-recovery-class failure (jumper + second computer), not SSH-fixable. xwing (Pi 3B+) has no EEPROM, doesn't share this exposure. This is why the kernel/bootloader/firmware tier needs to stay blacklisted regardless of what's decided for the rest of RPi Foundation origin.
- **One of the two original blockers turned out to already be resolvable, for free:** `/var/log/apt/history.log` is **world-readable** (`root:root 644`) on all three hosts — no `adm`/`systemd-journal` grant needed. It records every completed `unattended-upgrade` transaction with timestamps; checked 2026-07-14 and all three hosts show clean runs as recently as two days prior, confirming the service demonstrably works today. Nobody (me, security) had actually looked at this specific file before flagging "can we tell if it's running" as an unknown blocker — don't assume a verification gap is unresolvable just because the obvious path (`journalctl`, gated by `adm`/`systemd-journal`) is blocked; check for an alternate already-readable source before concluding "needs a privilege grant." The narrower residual gap (diagnosing an upstream systemd-level failure that never reaches a dpkg transaction) is tracked separately as `lucos_agent_coding_sandbox#99` (options laid out, no recommendation — group grant is still lucas42's call).
- **Remaining blocker, needs lucas42 directly:** physical accessibility of xwing/salvare for hardware recovery if a kernel or EEPROM update goes bad.
