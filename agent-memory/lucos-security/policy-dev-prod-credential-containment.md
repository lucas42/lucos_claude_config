name: policy-dev-prod-credential-containment
description: Dev environments must never hold a functioning prod-aithne (or any prod) credential, even a purpose-scoped one minted specifically for dev use
metadata:
  type: project
---

**Decision (2026-07-09, lucos_worlds dev-login blocker):** when lucos_creds' link-validation blocked linking dev worlds' `KEY_LUCOS_AITHNE` to aithne's production credential ("only read-only scopes... permitted on a link from non-production to production"), the proposed workaround — mint a *separate* dev-only OIDC client (`lucos_worlds_dev`) in prod aithne with its own secret + localhost redirect_uri, then hand-copy that secret into dev creds — was vetoed.

**Why:**
1. The lucos_creds link-validation only fires on *linking* dev to an *existing* prod credential. Hand-minting a fresh credential and manually writing it into dev creds never touches that check at all — and agents already have dev-environment write access, so nothing technical stops it. This degrades "dev never holds a working prod credential" from an enforced system invariant into a paperwork convention, which is a bigger loss than the specific credential's narrow scope suggests.
2. Residual risk on the credential isn't zero even with redirect_uri pinned to localhost — [[risk-prompt-injection-and-ci-logs]] and the broader "no internal trusted network" fact mean a compromised dev host or an accidentally-exposed dev port turns a leaked secret into live impersonation against prod aithne for that client.

**Round 2 (same day) — lucas42 proposed a genuinely different mechanism, also rejected, with a verified reason:** give the worlds→aithne creds link a scope, then allowlist that specific scope as an exception to the dev→prod guard (`allScopesReadOnly`, creds `storage.go:383`→`scopes.go:46`), so the link becomes a system-enforced, auditable exception rather than a hand-minted bypass. This is architecturally *better engineering* than the round-1 workaround (central, revocable, auditable) — but architect traced aithne's `handleAuthCodeGrant` (oidc.go:277-395) and **confirmed the scope would be inert**: the OIDC client is authenticated purely by secret-hash lookup (`GetOIDCClient`/`client.SecretHash`), and issued-token scopes come from the authenticating *principal* (`GetActiveScopes`), never from the creds-link's scope. So a scoped, allowlisted link would satisfy the guard's letter while conferring **zero** actual reduction in the credential's power — a leaked secret still lets the holder complete the client side of the auth-code exchange for any user, full stop.

**This is worse than the round-1 bypass, not equivalent to it.** A hand-minted bypass is visibly a bypass — anyone auditing it knows to scrutinise the exposure. A scoped-and-allowlisted link *reads* as governed in every future review, so a reviewer checking "does this credential have prod power in a dev environment?" sees the narrow scope and reasonably stops digging. Security theatre that degrades the *next* reviewer's judgement, not just this decision.

**General principle surfaced here:** scope-narrowing a *credential's link record* only reduces risk if the resource server actually enforces that link-level scope on the flow that credential grants. For OIDC client secrets specifically (and likely other bearer secrets authenticated by hash/value rather than by an accompanying claim), it may not — verify the enforcement point before crediting a scope with reducing blast radius, don't infer it from the vocabulary supporting scoped links.

**Resolution (final):** Option 3 (dev worlds → prod aithne, in any form) is dead. lucos_worlds dev uses standard (non-OIDC) auth for now to unblock local dev. The durable fix is a separate aithne-dev-infra initiative: mkcert-issued HTTPS cert in front of *dev* aithne, so dev satisfies OIDC clients' hard `https://` issuer requirements (e.g. BookStack's) without ever touching a prod credential. It's estate-wide (aithne's issuer derives from a single `APP_ORIGIN`, so serving dev aithne over HTTPS changes the issuer for *every* dev OIDC consumer and needs CA trust added to each consumer container), so it's tracked as its own initiative, not bundled into the lucos_worlds ticket. Architect was asked to add the scope-inertness finding to the aithne security architecture notes as a durable fact about that codebase.

**How to apply:** if any other lucos service hits this same "dev can't complete real OIDC/OAuth against a prod-only IdP" wall, the answer is the same: reject any path that ends with a working prod credential in dev — hand-minted separate client, or scoped/allowlisted creds link — regardless of how enforced or auditable the mechanism looks. Point them at (a) standard auth in dev as the cheap unblock, or (b) the aithne-dev-HTTPS initiative as the durable fix, once it exists. If someone proposes scoping a creds link to a *different* prod credential type as a middle ground, first confirm — don't assume — that the resource server enforces link-level scope on the flow in question.
