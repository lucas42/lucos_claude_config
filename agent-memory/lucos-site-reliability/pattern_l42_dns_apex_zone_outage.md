**Symptom:** every `*.l42.eu` service name SERVFAILs from public resolvers (8.8.8.8 AND 1.1.1.1), but `*.s.l42.eu` (host subdomain) still works → the `l42.eu` **apex** zone failed to load on the primary; `s.l42.eu` and other zones loaded fine. First seen 2026-06-07 (~23:34–23:48Z), a ~14-min total estate outage. Incident report: lucos `docs/incidents/2026-06-07-estate-wide-dns-outage.md`.

**Why / How to apply (diagnostic path):**
- Primary nameserver = `dns.l42.eu` = **avalon, 178.32.218.44**. Secondary = `dns2.l42.eu` = **xwing, 152.37.104.10**. `.eu` parent delegation for l42.eu = {dns.l42.eu, ns1.lukeblaney.co.uk} — NOT dns2.
- Query authoritative DIRECTLY by IP to bypass your own/public recursion: `dig l42.eu SOA @178.32.218.44`. SERVFAIL from the authoritative server for its OWN zone = zone not loaded (file error / expired / cold start). `dig version.bind CH TXT @<ip>` confirms BIND itself is alive.
- Confirm with logs: `docker logs lucos_dns_bind | grep -iE "l42.eu/IN|not loaded|loaded serial"`. Look for `dns_master_load: .../l42.eu:NN: <name>: CNAME and other data` → `not loaded due to errors`.
- 2026-06-07 cause: generated `/etc/bind/generated-zones/l42.eu` had `dns2 IN A 178.32.218.44` **and** `dns2 IN CNAME xwing.s` (an A+CNAME collision from an in-flight dns2→xwing migration; stale file from OLD generator code). BIND rejects the WHOLE zone on that.

**Infra quirks that matter here:**
- **`rndc` is unusable from the container** ("connection to remote host closed"). Reload via **`docker exec lucos_dns_bind kill -HUP <named-pid>`** (named was PID 8). SIGHUP reloads config+zones cleanly.
- Zones are GENERATED by `lucos_dns_sync` (config-sync.py, in the lucos_dns repo, container `lucos_dns_sync`) from configy, written to shared volume `lucos_dns_generatedzones`, then `rndc reload <zone>` (NOT a container restart). Runs every */15. **CHICKEN-AND-EGG:** if l42.eu is down, config-sync can't resolve configy.l42.eu to regenerate → can't self-heal until you restore the apex manually.
- BIND keeps the last in-memory copy on a FAILED reload, but a fresh container START against a broken file = apex simply absent (no fallback). So the outage tips over on a deploy/restart, not on the bad generation alone.
- Fix-then-restore: edit the generated file (remove bad line) → `named-checkzone l42.eu <file>` → SIGHUP. config-sync later overwrites the file; current generator is FIXED (special-cases dns/dns2 to A/AAAA glue, never CNAME) so its regeneration is valid (verified: serial bumped, loaded clean at 23:57:45).

**The amplifier — secondary had NEVER worked (TSIG BADKEY), now FIXED:** xwing `lucos_dns_secondary_bind` got NOTIFYs over IPv4 but EVERY transfer failed `tsig indicates error` — primary signed `tsig-transfer`, secondary expected `lucos-tsig` (names must match; secret matched). Estate had ZERO redundancy → one host's zone error = total outage. FIXED by **lucos_dns#103** (merged 2026-06-08 00:03:41Z); verified `dig l42.eu SOA @152.37.104.10` → NOERROR serial matching avalon, xwing slaves all 5 zones. If transfers break again, FIRST check the TSIG key NAME matches both sides (`startup.sh`). Apex/generator hardening (validate-before-install + last-known-good) = **lucos_dns#104** (High). **External redundancy NOW achieved (2026-06-08):** registrar/.eu delegation fixed to {dns.l42.eu→178.32.218.44 avalon, dns2.l42.eu→152.37.104.10 xwing} with IPv4 glue for both; ns1.lukeblaney CNAME dropped. Independence verified: xwing answers l42.eu authoritatively (`aa` flag, serial matches), reachable via parent glue independent of avalon. **Residual (low-pri, lucos#227; lucos_dns#107 closed→points at #227):** .eu glue is IPv4-ONLY for both dns+dns2 (registrar wouldn't save IPv6 glue) → pure IPv6-only resolver on cold cache during avalon outage can't bootstrap via parent (rare, pre-existing, symmetric). `.uk`/`.co.uk` zones delegate to {dns,dns2} too. Gotcha: .eu needs dns2 registered as a GLUE HOST OBJECT (in-bailiwick) before attaching to delegation; .uk/.co.uk don't (out-of-bailiwick).

**Detection gap:** monitoring.l42.eu is itself inside the blast radius — DNS outage won't self-alert. This one was surfaced via a stalled auto-merge PR (the auto-merge workflow reads configy for supervised status; failed estate-wide during the outage — same root cause behind code-reviewer's lucos_repos#410 "configy flap").
