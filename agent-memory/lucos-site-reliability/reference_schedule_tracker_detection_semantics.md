**How a scheduled-job failure reaches monitoring (ADR-0004, lucas42/lucos#140):**
- Cron jobs POST `{system, job_name, frequency, status, message}` to `schedule-tracker.l42.eu/v2/report-status` (client pkg `lucos-schedule-tracker-pythonclient`, `job_name` REQUIRED, default frequency 86400s).
- schedule_tracker persists per-`(system, job_name)` state and **derives the check server-side**.
- Post-cutover, schedule_tracker's OWN `/_info` carries only the service's intrinsic health (I saw **0 job checks** there — don't conclude "nothing monitored"). The job checks are exposed at **`GET /jobs`** (queryable, no auth) and pulled into lucos_monitoring by **`fetcher_scheduled_jobs`**, attributed to the **owning `system`** row (e.g. `lucos_arachne`), NOT the `lucos_schedule_tracker` row.

**Check semantics (from a live `/jobs` entry, 2026-07-02):** `ok = (at least one of the N most-recently-finished runs succeeded, N=error_threshold) AND (most recent completion within the frequency-derived threshold)`. Metrics: `age` (secs since last completion), `errors` (consecutive errors since last success). **⇒ red requires error_threshold CONSECUTIVE failed runs, or no completion within the staleness window.** A single/intermittent failure followed by a success stays GREEN by design — "monitoring was green during transient failures" is CORRECT, not a gap, unless genuinely persistent.

**⚠️ FALSE-RECOVERY BUG (verified in `src/database.rb`, 2026-07-22, tracked lucas42/lucos_schedule_tracker#96):** `age` is computed from `last_success || last_error`, but `updateScheduleError` writes via `INSERT OR REPLACE` **omitting `last_success`** → SQLite nulls `last_success` on every error report → `age` then measures from the last *error*, not the last success. Consequence: a job that goes silent (reds correctly via the `age` branch), then **resumes but keeps failing**, flips GREEN on its first error report (recent error resets age to ≈0, and error_count is still < threshold) and stays green until error_count reaches the threshold. i.e. **staleness guards against "stopped running", NOT against "running-but-failing".** This produced the 2026-07-21 configy_sync ~69-min false-recovery (lucas42/lucos_creds#474). Corollary: a genuinely-silent job (never resumes) DOES red via age — "forgotten job reads green" is WRONG; the reads-green case is a *deleted* row. Until #96 lands, don't trust a schedule-tracker check's GREEN as proof a job is actually succeeding — confirm the actual run outcome.

**arachne specifics:** bulk `ingest.py` reports **19 separate per-source jobs** (`job_name=<source system>`, e.g. `lucos_eolas`, `lucos_contacts`, `inference`, ontologies) — a per-source 403 is NOT swallowed and is NOT masked by the unconditional end-of-run `job_name="ingestor"` heartbeat (that's a separate "cron ran without crashing" signal). This is a DIFFERENT mechanism from the #702 `failed_item_ingest` check on the ingestor `/_info`, which covers the **per-item webhook** path ONLY (keyed on `_last_failure_at` set by `server.py._increment_failure`, self-clearing on clean reconcile).
