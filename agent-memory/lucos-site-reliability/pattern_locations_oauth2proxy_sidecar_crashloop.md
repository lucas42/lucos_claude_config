lucos_locations human map-UI paths (/map, /owntracks/api|ws|view|static|utils) are fronted by an `lucos_locations_oauth2_proxy` sidecar via nginx (`lucos_locations_otfrontend`) `auth_request`, gated on aithne `locations:read` (PR #97, closed/merged 2026-07-09).

**Failure mode (2026-07-10 incident):** oauth2_proxy crash-loops (`Restarting`, exit=1, rising RestartCount) with `main.go:67 invalid configuration: missing setting: cookie-secret / client-secret`. nginx dutifully auth_requests a dead sidecar ‚Üí **500 on /map/**. Cause: lucas42-only prod creds never set before #97 auto-deployed ‚Äî `OAUTH2_PROXY_COOKIE_SECRET` + `KEY_LUCOS_AITHNE` (OIDC client secret via prod linked cred lucos_locations/production ‚áí lucos_aithne/production, tracked lucas42/lucos_locations#96).

**Why /_info stays GREEN through it (CORRECTED ‚Äî sysadmin domain review 2026-07-10):** /_info is served by a SEPARATE local `info_server.py` (Python, port 8080) inside the `otfrontend` container, at an nginx `location /_info` with NO `auth_request` (`auth_basic off; proxy_pass http://127.0.0.1:8080/_info`). It never touches oauth2-proxy OR otrecorder's health ‚Äî it synthesises its own checks (mosquitto TLS, location-freshness via otrecorder `/api/0/last`) and always 200s. (otrecorder = third-party `owntracks/recorder` image, NOT a Go/lucos app ‚Äî don't call it that.) So /_info here can't observe the sidecar outage BY DESIGN. NB this is NOT universal: many lucos services proxy /_info straight to the backend and WOULD be blinded by a front-proxy failure. Live case of [[feedback_healthcheck_depth_varies]] / [[pattern_info_endpoint_boundary]] ‚Äî verify the SIDECAR container state + probe the human path; never trust /_info to catch an auth-layer outage.

**Human/device split tell:** device publish `/owntracks/pub` ‚Üí **401** (separate basic-auth, healthy) and MQTT `:8883` OPEN, while human `/map/` ‚Üí 500. A 401 on the device path (not 500) confirms only the human auth layer is down and ingestion/data are intact.

**Fix is lucas42-only** (prod creds): set the two creds + redeploy, or fallback-revert #97. SRE can't restore ‚Äî restarting the proxy is futile without creds.

**Resolution (2026-07-10, ~51min outage):** coordinated CI redeploy, NOT host restarts ‚Äî a bare `docker restart` reuses the container's old env and won't inject new lucos_creds values; only a fresh deploy re-fetches creds. Order: (1) **aithne** first ‚Äî its `oidc_clients` reconcile from `CLIENT_KEYS` is **startup-only** (ADR-0004, `reconcileOIDCClients` called once from `main()`), so it must restart to register the new client; confirm via log `oidc client reconcile: upserted client "lucos_locations"`. Redeploying the estate auth SPOF for a linked-cred change is expected/normal (clean convergence). (2) **lucos_locations** ‚Äî inject the sidecar creds. Both via `POST /api/v2/project/gh/lucas42/<repo>/pipeline {"branch":"main"}`.

**Gated paths are STRUCTURALLY fail-closed (security-verified 2026-07-10).** When oauth2-proxy is down, nginx `auth_request` gets connection-refused on `/oauth2/auth`; the module returns 500 (deny) for any non-2xx/401/403 subrequest, and no `error_page` in otfrontend maps an auth_request failure to "allow". So a sidecar outage can never leak access ‚Äî 500 is the only possible outcome, not just what was observed once. (`/owntracks/api/0/kill` is also hard-denied 403 regardless of scope ‚Äî defence-in-depth.)

**STATUS 2026-07-14: fully resolved and stable, and #99/#100 have SHIPPED.** `lucos_locations_oauth2_proxy` `restarts=0 oom=false`, up since 2026-07-10 00:33, **zero error lines in 7d**. Live-probed the human path (the layer /_info cannot see): `/map/` ‚Üí **302** (the auto-redirect #100 intended ‚Äî the 403-sign-in-page behaviour below is now HISTORY), `/oauth2/start` ‚Üí 302 to aithne with correct `client_id=lucos_locations` + `scope=openid email profile locations:read` + redirect_uri, device `/owntracks/pub` ‚Üí **401** (basic-auth intact), `/_info` ‚Üí 200. Residual: the sidecar has **no healthcheck** (`health=none`), so the /_info blind spot is structural ‚Äî but there's no evidence of impact, so no issue filed (fails impact-vs-effort).

**`/map/` unauth returned 403 (a sign-in PAGE), not a 302 ‚Äî was BY DESIGN at incident time (2026-07-10), now superseded by the 302 above.** #97's otfrontend nginx used `error_page 401 =403 /oauth2/sign_in` on gated locations: the auth_request 401 rewritten to 403, served the oauth2-proxy sign-in-button page (button ‚Üí /oauth2/start ‚Üí 302 ‚Üí aithne). So a headless `/map/` probe returning 403-with-sign-in-HTML was HEALTHY, not broken. **CHANGING (lucos_locations#99/#100, lucas42 chose "split" 2026-07-10):** `/map` will AUTO-REDIRECT (302) straight to aithne, while the API/WS paths (`/owntracks/api|ws|‚Ä¶`) KEEP 401/403 (a 302 there breaks fetch/WS). So post-#100: `/map` 302, API/WS still 401/403. Either way, verify OIDC wiring via `/oauth2/start` ‚Üí 302 (correct client_id/scope/redirect_uri), not via `/map/` status code alone.
