lucos_firewall's generated `DOCKER-USER` chain (main.go `generateIPv4Ruleset`/`generateIPv6Ruleset`) ends in a terminal `-j DROP` with allow rules qualified by **protocol+dport only** (no `-i`/`-o`/source). This means it polices **all** forwarded traffic, including container-to-container ‚Äî broader than ADR-0007's stated scope ("inter-container controls stay on the Compose-network model"). Confirmed by reading source at `b1296b5` (2026-06-08).

Key mechanism facts (reusable):
- **Docker evaluates DOCKER-USER first**, before its own same-bridge ICC ACCEPT ‚Äî documented contractual purpose of the chain, not insertion-order luck. So a terminal DROP in DOCKER-USER wins over Docker's container-traffic ACCEPTs. Don't expect Docker's ICC rule to "save" inter-container traffic.
- `ESTABLISHED,RELATED -j ACCEPT` rescues only **return** traffic of already-established flows ‚Äî never the initial NEW SYN of a fresh inter-container connection.
- **Cross-network (different bridges):** always L3-routed ‚Üí always hits FORWARD‚ÜíDOCKER-USER ‚Üí dropped under enforce if dest port not public. Definite.
- **Same-bridge (same compose stack, e.g. app‚Üîits own Postgres):** traverses FORWARD only if host `bridge-nf-call-iptables=1`. If set, enforce breaks app‚Üîdatabase estate-wide. **Check this sysctl on each host before any enforce flip** ‚Äî it sizes the blast radius.
- Dry-run does NOT surface this (generates ruleset, drops no packets) ‚Äî latent until enforce.
- Converse oddity: declaring a port `public` currently also opens it inter-container (allow rule is origin-blind).

**RESOLVED in firewall#14 (merged 2026-06-08):** added `-A DOCKER-USER -i br+ -j RETURN` + `-i docker0 -j RETURN` after the ESTABLISHED/RELATED ACCEPT, before the public-port allow-list. This RETURNs **all** bridge-origin traffic before the DROP ‚Äî so the firewall now polices **external-origin traffic only**.

Key correction to my original analysis above: `-i` matches **ingress** interface = the **source** container's bridge, regardless of destination. So the RETURN exempts **same-stack AND cross-stack** container traffic alike (cross-stack A`br-aaaa`‚ÜíB`br-bbbb` is *received* on `br-aaaa`, matches `br+`). The firewall does **not** isolate Compose stacks from each other ‚Äî only Docker's own ICC/isolation + app-level auth do. My earlier lucos_dns#106 conclusion (cross-bridge config-sync‚Üíconfigy would be dropped) is **reversed** by #14: it would now survive enforce. (Cache fix still the right #106 resolution.)

ADR-0007 Amendment 2 (PR #229) documented #14 but mis-stated the cross-stack consequence as "still dropped" (contradicting its own Decision section) and left a pre-existing `:FORWARD DROP` skeleton error (impl uses `:FORWARD ACCEPT` ‚Äî Docker owns FORWARD; default-deny lives in INPUT policy + DOCKER-USER terminal DROP). **GENERALISED PATTERN (frame-review for any future firewall change):** ADR-0007's default-deny has now over-reached beyond its *internet-facing* scope **twice** ‚Äî Amendment 2 (DOCKER-USER terminal DROP caught bridge inter-container traffic) and Amendment 3 (INPUT DROP caught link-local mDNS UDP 5353, severing `.local` resolution + the backups‚Üíaurora poll, 2026-06-08). Both fixes are *scope-corrections* (restore intended internet-facing scope), not loosenings. **When reviewing any firewall rule, ask: does this DROP catch link-local / inter-container / multicast / other non-internet traffic the ADR never meant to police?** Link-local infra (ICMP, ICMPv6, NDP, mDNS) needs explicit base-allow carve-outs because the default-deny is origin-blind. Amendment 3 = link-local mDNS base-allow scoped to multicast groups `224.0.0.251`/`ff02::fb` (internet-unreachable by construction); impl in lucos_firewall#16, draft ADR PR lucos#233 (unsupervised ‚Üí draft + @lucas42 ping, awaiting sign-off). Naming-policy corollary recorded: managed unicast DNS (`*.s.l42.eu`) is authoritative; mDNS `.local` is human convenience, NOT an automation dependency ‚Äî aurora is the lone host on `.local`, moving off it tracked in lucos_backups#306 (homes the lucos_backups ADR-0001 addressing-omission). `.local`=mDNS multicast (firewall-fragile); `.lan`=router unicast (firewall-safe, reply rides ESTABLISHED/RELATED).

**FOURTH over-reach instance ‚Äî host-networked router-fronted service (2026-06-08, lucos_backups#307):** avalon enforce dropped `backups.l42.eu/_info`. Root cause: `lucos_backups` is `network_mode: host`, so the router's hairpin to it is **host-destined ‚Üí INPUT chain**, and the `-i br+/docker0 -j RETURN` bridge-hairpin exemption lives ONLY in DOCKER-USER (FORWARD), not INPUT. backups is the **only** host-networked router-fronted service estate-wide. **Key principle: host-networking collapses the backend/published-port distinction the firewall's model depends on** ‚Äî for bridged services backend-closed and router-reachable live on different planes (FORWARD hairpin, #14-exempted); for host-net services they collide on the same host port/interface. **Resolution chosen: remove the host-networking (bridge the service, #307), NOT amend the firewall** ‚Äî once bridged it's covered by #14, no firewall change, no exposure, no INPUT special-case. Rejected: declare-port-public (re-opens backend port on public iface; security-negative). Fallback only: port-scoped INPUT bridge-hairpin RETURN ‚Äî but INPUT bridge-RETURN is container‚Üíhost (stronger trust boundary than #14's container‚Üícontainer), so must be port-scoped not blanket. **Reusable insight (true): a container reaching a `.local` host via SSH ProxyJump does NOT itself need mDNS/host-net** ‚Äî paramiko/fabric `gateway=` resolves the final-hop hostname on the JUMP host (xwing), not the client. **BUT my conclusion "backups' host-net was vestigial" was WRONG** ‚Äî lucas42 corrected via commit `bb81f37` (Oct 2024): host-net was added for **IPv6 outbound to salvare** ("salvare only available over ipv6"), nothing to do with aurora. LESSON: I inferred the *reason* for a config from one plausible code angle without `git log`-ing *why the mode was added* ‚Äî and ADR-0007 ¬ßContext even documented the IPv6‚Üísalvare reason, which I missed. **Check the commit that introduced a setting before theorising its purpose.** Option 3 still survives, but my SAFETY EVIDENCE was wrong TWICE (SRE corrected): I claimed `lucos_monitoring`+`lucos_time` run bridged with `enable_ipv6` (fd00:1/2) ‚Äî FALSE, I read their *compose files* (declared enable_ipv6) as *running* networks; their live nets are IPv4-only (`EnableIPv6=false`, never recreated) and fd00:2 is `lucos_dns_default`'s. I also claimed monitoring fetches salvare /_info bridged ‚Äî FALSE, salvare serves no HTTP. **Real verified precedent = `lucos_dns_bind`**: runs bridged on an enable_ipv6 ULA net and reaches salvare's GUA (NAT66 working), + host-net backups‚Üísalvare:22 CONNECT_OK today. Corrected impl: remove host-net **AND add `enable_ipv6` net `fd00:3::/64`** ‚Äî a bare removal WOULD break salvare. Gate sharpened: verify IPv6‚Üísalvare from bridged container before merge ‚Äî and the gate caught this real dep before any change. Interim while #307 pending: hold (backups function fine under enforce post-Amendment-3; only inbound /_info/web down = monitoring-visibility gap, P2); keep avalon enforced, silence the backups monitoring check, no public-port stopgap.

**FIFTH instance ‚Äî whole-table flush wipes Docker chains (lucos_firewall#19, 2026-06-08):** firewall `runRestore` pipes the ruleset to `iptables-restore` **without `--noflush`** = whole-table `*filter` replace declaring only INPUT/FORWARD/DOCKER-USER ‚Üí deletes ALL Docker-owned filter chains incl. Docker 28+/29's **`DOCKER-FORWARD`**. Existing traffic survives (nat table intact + FORWARD ACCEPT), but `docker network create` runs `iptables -A DOCKER-FORWARD ‚Ä¶` ‚Üí fails ‚Üí **new network creation broken on all 3 enforced hosts** (estate regression; also blocks #307's bridge net). **Fix (architectural direction posted):** switch to `--noflush`; pre-flush ONLY the two chains we own (`iptables -N DOCKER-USER 2>/dev/null; -F DOCKER-USER; -F INPUT`); drop the `:FORWARD ACCEPT` whole-chain policy-replace, but KEEP our `FORWARD -j DOCKER-USER` jump made **idempotent** (`-C || -I FORWARD 1`) ‚Äî do NOT rely on Docker maintaining its own jump to make our DROP reachable (that's the coupling that keeps biting); never reference DOCKER/DOCKER-FORWARD/DOCKER-ISOLATION. **DURABLE PRINCIPLE (now 4 amendments deep): lucos_firewall owns exactly two filter-table things ‚Äî host INPUT policy+allow-list, and DOCKER-USER content ‚Äî and touches nothing else. Whole-table replace is the recurring root cause; --noflush+own-chains-only makes it version-decoupled from Docker's chain layout.** Minor atomicity trade-off (sub-second empty-INPUT window; conntrack+TCP-retransmit covers it; optional custom-chain atomic swap if zero-window needed).
