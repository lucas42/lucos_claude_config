lucos_firewall's generated `DOCKER-USER` chain (main.go `generateIPv4Ruleset`/`generateIPv6Ruleset`) ends in a terminal `-j DROP` with allow rules qualified by **protocol+dport only** (no `-i`/`-o`/source). This means it polices **all** forwarded traffic, including container-to-container — broader than ADR-0007's stated scope ("inter-container controls stay on the Compose-network model"). Confirmed by reading source at `b1296b5` (2026-06-08).

Key mechanism facts (reusable):
- **Docker evaluates DOCKER-USER first**, before its own same-bridge ICC ACCEPT — documented contractual purpose of the chain, not insertion-order luck. So a terminal DROP in DOCKER-USER wins over Docker's container-traffic ACCEPTs. Don't expect Docker's ICC rule to "save" inter-container traffic.
- `ESTABLISHED,RELATED -j ACCEPT` rescues only **return** traffic of already-established flows — never the initial NEW SYN of a fresh inter-container connection.
- **Cross-network (different bridges):** always L3-routed → always hits FORWARD→DOCKER-USER → dropped under enforce if dest port not public. Definite.
- **Same-bridge (same compose stack, e.g. app↔its own Postgres):** traverses FORWARD only if host `bridge-nf-call-iptables=1`. If set, enforce breaks app↔database estate-wide. **Check this sysctl on each host before any enforce flip** — it sizes the blast radius.
- Dry-run does NOT surface this (generates ruleset, drops no packets) — latent until enforce.
- Converse oddity: declaring a port `public` currently also opens it inter-container (allow rule is origin-blind).

**RESOLVED in firewall#14 (merged 2026-06-08):** added `-A DOCKER-USER -i br+ -j RETURN` + `-i docker0 -j RETURN` after the ESTABLISHED/RELATED ACCEPT, before the public-port allow-list. This RETURNs **all** bridge-origin traffic before the DROP — so the firewall now polices **external-origin traffic only**.

Key correction to my original analysis above: `-i` matches **ingress** interface = the **source** container's bridge, regardless of destination. So the RETURN exempts **same-stack AND cross-stack** container traffic alike (cross-stack A`br-aaaa`→B`br-bbbb` is *received* on `br-aaaa`, matches `br+`). The firewall does **not** isolate Compose stacks from each other — only Docker's own ICC/isolation + app-level auth do. My earlier lucos_dns#106 conclusion (cross-bridge config-sync→configy would be dropped) is **reversed** by #14: it would now survive enforce. (Cache fix still the right #106 resolution.)

**GENERALISED PATTERN (frame-review for any future firewall change):** ADR-0007's default-deny has now over-reached beyond its *internet-facing* scope **twice** — Amendment 2 (DOCKER-USER terminal DROP caught bridge inter-container traffic) and Amendment 3 (INPUT DROP caught link-local mDNS UDP 5353, severing `.local` resolution + the backups→aurora poll, 2026-06-08). Both fixes are *scope-corrections* (restore intended internet-facing scope), not loosenings. **When reviewing any firewall rule, ask: does this DROP catch link-local / inter-container / multicast / other non-internet traffic the ADR never meant to police?** Link-local infra (ICMP, ICMPv6, NDP, mDNS) needs explicit base-allow carve-outs because the default-deny is origin-blind. Amendment 3 = link-local mDNS base-allow scoped to multicast groups `224.0.0.251`/`ff02::fb` (internet-unreachable by construction); impl in lucos_firewall#16, draft ADR PR lucos#233 (unsupervised → draft + @lucas42 ping, awaiting sign-off). Naming-policy corollary recorded: managed unicast DNS (`*.s.l42.eu`) is authoritative; mDNS `.local` is human convenience, NOT an automation dependency — aurora is the lone host on `.local`, moving off it tracked in lucos_backups#306 (homes the lucos_backups ADR-0001 addressing-omission). `.local`=mDNS multicast (firewall-fragile); `.lan`=router unicast (firewall-safe, reply rides ESTABLISHED/RELATED).

**FOURTH over-reach instance — host-networked router-fronted service (2026-06-08, lucos_backups#307):** avalon enforce dropped `backups.l42.eu/_info`. Root cause: `lucos_backups` is `network_mode: host`, so the router's hairpin to it is **host-destined → INPUT chain**, and the `-i br+/docker0 -j RETURN` bridge-hairpin exemption lives ONLY in DOCKER-USER (FORWARD), not INPUT. backups is the **only** host-networked router-fronted service estate-wide. **Key principle: host-networking collapses the backend/published-port distinction the firewall's model depends on** — for bridged services backend-closed and router-reachable live on different planes (FORWARD hairpin, #14-exempted); for host-net services they collide on the same host port/interface. **Resolution chosen: remove the host-networking (bridge the service, #307), NOT amend the firewall** — once bridged it's covered by #14, no firewall change, no exposure, no INPUT special-case. Rejected: declare-port-public (re-opens backend port on public iface; security-negative). Fallback only: port-scoped INPUT bridge-hairpin RETURN — but INPUT bridge-RETURN is container→host (stronger trust boundary than #14's container→container), so must be port-scoped not blanket. **Reusable insight: a container reaching a `.local` host via SSH ProxyJump does NOT itself need mDNS/host-net** — paramiko/fabric `gateway=` resolves the final-hop hostname on the JUMP host (xwing), not the client; so backups' host-net was likely vestigial. (Hedge: runtime-verify before dropping the mode — possible hidden host-net dep, e.g. reaching the local host's own services.) Interim while #307 pending: hold (backups function fine under enforce post-Amendment-3; only inbound /_info/web down = monitoring-visibility gap, P2); keep avalon enforced, silence the backups monitoring check, no public-port stopgap.
