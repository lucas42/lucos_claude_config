# DNS secondary modelling review (2026-06-05)

lucas42 halted the secondary-DNS work (lucos_dns#79, PR lucos_dns#95, configy PR #208) over two modelling problems. I reviewed; both well-founded, one root cause.

**Root cause:** configy assumes **one system → one domain → one host**. Load-bearing in: `lucos_dns/sync/config-sync.py` (`hosts[0]` for the `dns` A record), `/systems/subdomain/{root}` endpoint, and the `domain.is_some() ⇒ hosts.len()<=1` validation invariant. Homogeneous multi-host systems (lucos_firewall, lucos_docker_health) fit because they have **no domain** and run identically per host, keyed only on `HOSTDOMAIN`. DNS secondary is **heterogeneous**: same image, two roles (primary/secondary), two hostnames (dns/dns2.l42.eu), per-host config — the model doesn't support it.

**Why concern #1 (configy PR #208) is right:** gating the invariant on `http_port` couples "speaks HTTP" to "may be multi-homed on a domain" (unrelated). Makes configy data false (asserts lucos_dns serves dns.l42.eu on xwing; xwing serves dns2). Has no functional effect — config-sync uses `hosts[0]` + a **hardcoded `'xwing' in host_domain_lookup` special-case** for dns2 glue, not the hosts list. PR #208 only satisfies the `circleci-system-deploy-jobs` check.

**Why concern #2 (PR #95) is right — fatal flaw:** deploy orb keys the prod envfile on `CIRCLE_PROJECT_REPONAME` (`deploy.yml`: `scp …:$CIRCLE_PROJECT_REPONAME/production/.env`). Both deploy-avalon and deploy-xwing read the SAME `lucos_dns/production/.env`. **Only per-host var injected is `HOSTDOMAIN`** (`export HOSTDOMAIN=<<host-domain>>`) — exactly lucas42's named exception. So `DNS_MODE`/`COMPOSE_PROFILES` differing per host is impossible. `TSIG_SECRET` is fine (same value both sides).

**Recommendation — two systems, one repo, one image:** distinct system codes `lucos_dns` (avalon/dns.l42.eu/primary) + secondary (xwing/dns2.l42.eu), both from lucos_dns repo / lucas42/lucos_dns_bind image. Each gets own creds namespace ⇒ per-host config for free. Restore (don't weaken) the invariant — revert #208's validation change. dns2 becomes a real configy domain ⇒ config-sync drops the xwing hardcode + hosts[0] fragility. Redundancy modelled at DNS protocol layer (two NS, two failure domains).

**Real cost:** one repo can't deploy as two system codes today (creds path is REPONAME-keyed). Needs a new orb `deploy` param (likely `creds-system`, default `CIRCLE_PROJECT_REPONAME`) so the secondary job fetches its own namespace while pulling the shared image. This is why it's an ADR, not a PR fix.

**Alternatives recorded:** (1) role-from-HOSTDOMAIN in one system — no pipeline change but hardcodes topology into image, promotion=image change, leaves dns2 problem; rejected. (2) generalise configy to per-host roles `hosts:[{host,role,domain}]` — most expressive but touches every consumer for a 2-instance case; deferred until a 3rd heterogeneous multi-host system appears. (3) lucas42's: **two separate repos+images** — rejected after commonality analysis (#213 comment): only `lucos_dns_bind` is shared (sync is already primary-only via `profiles:[primary]`); bind image is 100% shared Dockerfile+named.conf, same 5-zone list, divergence is only per-zone `type primary` vs `type secondary` (same BIND, slaving vs authoring). Two repos duplicate the zone list (silent-divergence failure mode), force 2-repo flag-days, risk BIND version skew, and don't even remove the `DNS_MODE` conditional (just relocate it). One-repo fix: replace the two heredocs with a single zone-list loop. Two-repos only right if secondary used different DNS software / grew secondary-only logic / different trust domain — none hold.

**Vehicle:** ADR raised at **lucas42/lucos#213** (proposed home lucas42/lucos, sibling to ADR-0003 / lucos#110; flagged configy as alt home). Assessment comment on lucos_dns#79. All three implementation artifacts on hold pending the ADR. Owner: lucos-architect.

**Position shift (2026-06-05, after lucas42 pushback):** my initial "high commonality → clearly two-systems-one-repo" was overstated. Corrected ledger (#213 comment): once you discount generic BIND boilerplate, the primary-only sync image, physically-shared-but-inert config (rndc/port 953 — only sync uses rndc; secondary reloads via AXFR/NOTIFY), `/etc/bind/zones/` (inert on secondary), and TSIG (a creds artifact, not repo-shared), the *usefully* shared surface is thin: generic BIND boilerplate + the 5-zone list. The ONLY surviving one-repo argument is the 5-zone list as single source of truth (silent slave/author divergence if split) — but it changes ~once per few years, so low-frequency/mitigable. Two repos is NOT wrong; per-repo clarity is a real counterweight. **The deploy-pipeline + configy modelling conclusions (two systems, separate creds namespaces, revert configy#208 validation, dns2 as real domain) STILL HOLD regardless of repo split** — repo structure is a separate axis from system modelling. **DECIDED 2026-06-05 (lucas42): two repos / two systems.** Deciding factor = estate-wide `repo == system` invariant, all keyed on `CIRCLE_PROJECT_REPONAME`: creds path (`$REPONAME/production/.env`), monitoring suppress (`/suppress/$REPONAME`), Loganne component/systemDeployed, lucos_repos `RepoTypeSystem` audit. Two-systems-one-repo would break all four (needs a `system-name` override threaded through the shared orb) — disproportionate for one secondary. Two repos = zero shared-tooling change AND cleanest data model: two normal single-host systems, **configy#208 superseded not reworked** (existing `domain⇒hosts<=1` rule holds), generic dns2 glue (drop the `'xwing'` hardcode), no orb param, no DNS_MODE branch. Only loss: 5-zone list duplicated across 2 repos. **Open sub-decision:** hardcode-the-5-zones+runbook (architect lean, proportionate) vs BIND catalog zone (cleaner, future-proofs growth, adds moving part, verify BIND≥9.16). NOT a convention check (cross-repo content-equality ≠ lucos_repos per-repo audit model). Resulting work itemised in #213 decision comment (new repo lucos_dns_secondary; new configy entry; close #208; trim #95 to primary-side only; config-sync generic glue; TSIG in both creds namespaces; registrar glue lucos#111). Tickets to be filed once lucas42 confirms direction + sub-decision; ADR to draft then (home lucas42/lucos, lighter than first scoped).

**MONITORING MECHANISM CORRECTED 2026-06-05** (lucas42 flagged: DNS boxes have no HTTP server, so `/_info` framing was wrong). Verified + SRE-confirmed: lucos_dns/lucos_dns_secondary are **excluded from `/_info` polling** — lucos_monitoring's `info-systems-list.json` is built at Docker-build time from configy `/systems/http` (http_port-only); live, lucos_dns shows only `[config-sync, circleci]`, **no fetch-info**. So #217's "fetch-info covers secondary-down" was FALSE. **Decided mechanism:** secondary-side **cron → curl POST to schedule_tracker `/v2/report-status`** (no python client; lucos_dns_sync config-sync push is the reference), consumed by monitoring's `fetcher_scheduled_jobs` (keys off job's `system` field, graceful fallback surfaces non-HTTP systems). One **aggregate** job `soa-serial-in-sync` (dig all 5 zones' SOA local @localhost vs primary @dns.l42.eu; error if any behind; per-zone serials in message). **~5-min cadence** — schedule_tracker derives grace from frequency (NOT a free knob): grace≈5/4/3/2 for <10m/<30m/<90m/≥90m, overdue=freq×3; 5min → ~25min error-grace + ~15min overdue. **Don't use ≥90min** (grace collapses to 2). Job MUST run on secondary so overdue-covers-down. No deploy-window suppression on scheduled-job checks (only fetch-info gets it) — grace absorbs it. **Artifacts:** #217 rewritten (title+body) to this mechanism, then **transferred to lucos_dns_secondary#2** (was lucas42/lucos#217); SRE-confirmed aggregate (one primary/TSIG/AXFR/cron ⇒ moves as a unit) + 4 comparison-correctness criteria folded in: (a) empty/missing local serial ⇒ FAIL not pass, (b) primary-unreachable ⇒ distinct outcome not "behind", (c) assume date-based serials/document/no RFC1982, (d) operator-legible per-zone message; lucos_dns_secondary#1 `/_info` furniture item struck (off hold); ADR-0010 follow-up+positive bullets corrected via draft **PR lucos#218** (pending lucas42 nod). **Reusable lesson:** "expose via /_info" assumes an HTTP server — verify the service HAS one before specifying /_info; BIND/cron/non-HTTP workloads use schedule_tracker push (fetcher_scheduled_jobs), which monitoring consumes equally.
