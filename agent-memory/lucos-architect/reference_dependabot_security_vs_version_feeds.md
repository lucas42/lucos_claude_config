name: Dependabot security updates are independent of dependabot.yml schedule
description: Reference for advising on Dependabot config — version updates and security updates are separate feeds with different triggers
type: reference

Dependabot has two independent update feeds, easily confused:

- **Version updates** — driven by `dependabot.yml`. Governed by `schedule.interval` (daily/weekly/monthly), `groups`, `allow`, `ignore`, etc. This is the feed that produces routine churn.
- **Security updates** — driven by GitHub Security Advisory matches against the repo's dep graph. Event-triggered, fires when an advisory is published. **Independent of `schedule.interval`** — moving the version-update schedule to weekly does not delay security patching.

Verbatim from GitHub docs (["About Dependabot security updates"](https://docs.github.com/en/code-security/dependabot/dependabot-security-updates/about-dependabot-security-updates)):

> "There is no interaction between the settings specified in the `dependabot.yml` file and Dependabot security alerts, other than the fact that alerts will be closed when related pull requests generated by Dependabot for security updates are merged."

## Practical implications

- **Slowing routine churn is safe.** `schedule.interval: weekly` (or `monthly`) does not delay patching for known vulnerabilities. Security PRs continue arriving on their own cadence.
- **Per-repo setting required.** Dependabot security updates are a separate per-repo toggle from Dependabot alerts and from version updates. Verify it's enabled before relying on the security feed for prompt patching.
- **Suppress version, keep security.** Setting `open-pull-requests-limit: 0` on an ecosystem suppresses version updates entirely while leaving security updates intact. Verbatim from GitHub: *"If you only require security updates and want to exclude version updates, you can set open-pull-requests-limit to 0 in order to prevent version updates for a given package-ecosystem."* Useful for repos in archive/maintenance mode.
- **Some `dependabot.yml` settings DO bleed into security PRs** — assignees, reviewers, target-branch, labels. So a misconfigured assignees list affects both.

## When to cite

- Anyone proposing changes to Dependabot frequency/grouping who is reasonably worried about vuln response time.
- Reviews that flag "this repo doesn't have version updates configured" — confirm whether security updates are nonetheless on.
- ADRs covering CI/dep-management cadence trade-offs.
