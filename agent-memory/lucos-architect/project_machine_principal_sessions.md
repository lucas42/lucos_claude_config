# Machine-principal sessions (lucas42/lucos#132 auth design)

Design agreed with lucas42 2026-06-01 while framing how lucos_root#135 (live homepage dashboard) depends on the new auth service (lucas42/lucos#132). Tickets are Ideation — this is forward design, not yet built.

**Decision:** the new auth service should let a **non-human principal** (a service like lucos_root, or a trusted LLM agent) obtain a session **non-interactively** (pre-shared key, not a passkey ceremony), and backends validate that session **identically to a human session** (same `/data`-style lookup, resolving to a service-account contact ID).

**Why:** the alternative — a parallel bearer scheme — forces **dual authentication** on every backend (each grows a second auth path), because the estate's existing SSE / web-component endpoints are all built around the human session. Making machine principals session-capable means backends need **zero auth change**: lucos_root presents the session as a cookie (it's a server-side client, can set any header) and consumes existing human-session SSE feeds unchanged. Separate *acquisition* (non-interactive) from *presentation* (ordinary session) — the earlier wish-list discussion conflated them.

**Unification:** "lucos_root reads estate data as a service" and "trusted LLM agent authenticates" are the same requirement — a non-human principal needs a session. Solve once.

**Critical caveat — unify authN, NOT authZ:** a machine session validates like a human session but must NOT *be able to do* everything a human can. Any backend gating a sensitive action on "is there a valid session?" (not on contact ID) would start accepting machine principals. Two guard rails, decided in #132 not per-backend: (1) machine sessions scoped least-privilege; (2) session-validation response distinguishes principal class (human / lucos_root / agent) via contact ID or principal-type claim, for per-backend differentiation + clean audit.

**Lifecycle:** the non-interactive acquisition key mints sessions → crown-jewel secret → lucos_creds, per-env, rotatable. Larger blast radius than a normal service key (reinforces least-privilege scoping).

**Acquisition mechanism (resolved with lucas42 2026-06-02, #132):** lucos_creds is the home for the *acquisition key* — and it already does this (CI/CD deploy, agents via lucos_agents entry), so no new mechanism. lucas42's caveat "creds is deploy-time-pull-only, so unsuitable for short-lived tokens" is correct but lands on the *session* layer, which we deliberately keep OUT of creds: creds stores the long-lived key (deploy-injected into the principal's env), the short-lived session is minted at runtime by exchanging that key against the auth service, never stored in creds, never pulled at deploy. So no runtime creds dependency anywhere on the hot path — and that local-key-holding is what makes dropped/expired SSE streams cheap to re-mint+reconnect. Shape = OAuth2 client-credentials grant (key ≈ client credential, session ≈ access token), which satisfies the "standard protocol" wish-list item. Validator side symmetric: auth service holds a CLIENT_KEYS-style registry of valid keys, also creds-provisioned at its deploy. Rotation = same drill as Bearer migration (dual-accept window, takes effect next deploy).

**How to apply:** when #132 is designed/implemented, hold the line that the machine path yields a *session*, not a bearer scheme, and that authZ stays separate from authN. This is what makes lucos_root#135 cheap (no backend touches its auth). See also [[reference_service_worker_ui_as_system_component]] for the dashboard-compositor shape.

**Centralised rate-limiting (resolved with lucas42 2026-06-03, #132):** lucas42 asked whether per-service rate-limiting (CodeQL/security flagging brute-force/resource-exhaustion on each backend's auth integration) can be centralised by redirecting unauthenticated users to a central system. Answer: decompose into (1) login ceremony — auth service only; (2) per-request session validation — today each backend calls `/data?token=` per request = amplification + the "validate credential against store" pattern CodeQL flags; (3) generic volumetric DoS. **The "by necessity each backend must check" pessimism is true ONLY in today's topology (open ports = central redirect is theatre, attacker hits port directly).** The firewall (ADR-0007, priority #1) inverts this: once enforcing, backend `http_port`s leave `public_ports` → reachable only via router → forward-auth at the router (nginx `auth_request`) becomes both possible AND safe, centralising redirect+check+rate-limit. **Footgun: trusted-header (`X-Lucos-Contact-Id`) injection is an estate-wide auth bypass if built before firewall enforce — strictly downstream of enforce.** Independent now-win regardless of firewall/gateway: **locally-verifiable signed sessions (JWKS)** — backend verifies signature locally, no outbound `/data` call → kills amplification + brute-force framing; same token the OAuth2 client-credentials machine path issues. Login/mint rate-limiting can never push down but lives on one system (auth service) by construction; volumetric DoS = router `limit_req` / Layer-3 (deferred in ADR-0007).
