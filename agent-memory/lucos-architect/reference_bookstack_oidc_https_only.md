BookStack's OIDC integration **hard-requires HTTPS** for the issuer and every
discovered endpoint (auth/token/userinfo), with no config toggle to relax it.
- `OidcProviderSettings::validateInitial()`: `if (!str_starts_with($issuer, 'https://')) throw InvalidArgumentException('Issuer value must start with https://')` — fires before any network call.
- `validate()` enforces the same `https://` rule on token/auth/userinfo endpoints; server-side calls use `CURLOPT_SSL_VERIFYHOST` (real cert verification, so self-signed fails without trust-store install).

**Symptom:** local-dev lucos_worlds login shows generic "An unknown error occurred". Real error in `/config/log/bookstack/laravel.log`: `Issuer value must start with https://`. Cause: dev aithne serves plain `http://localhost:8039`, so `OIDC_ISSUER=http://localhost:8039` is rejected instantly. **Dev-only** — prod aithne is `https://aithne.l42.eu`, login works.

Consequences for any local-dev OIDC handshake against BookStack: genuine local testing needs a TLS-terminating proxy in front of aithne with a locally-trusted CA (mkcert) installed into *both* the host browser and the `lucos_worlds_web` container trust store. Cheaper path for routine dev: `AUTH_METHOD=standard` + seeded local admin; exercise real OIDC in prod/staging.

Latent secondary gap: `lucos_worlds_web` compose lacks `extra_hosts: "host.docker.internal:host-gateway"` (aithne has it), so the container can't reach the host by name — only by raw gateway IP. Bites only once the https barrier is cleared. See [[reference_external_access_to_lan_host]]. (Raised as lucas42/lucos_worlds#36 — low/latent.)

**2026-07-09 decision (lucos_worlds#35):** lucas42's "Option 3" (point dev worlds issuer at *prod* aithne `https://aithne.l42.eu`) is **blocked by a lucos_creds guardrail** — a non-prod env may only link to a prod credential when the scope is `:read`; the OIDC client secret isn't read-only, so dev can't hold it (`Validation Error: only read-only scopes ... permitted on a link from non-production to production`). The guardrail is correct: dev must not hold a working prod-aithne secret. A *separate* hand-minted dev OIDC client is NOT a way round it (security concur): minting+hand-copying a secret bypasses the link-validation mechanism entirely (agents have dev write access), degrading an enforced invariant to a convention; and there's no internal trusted network so "localhost redirect_uri only" is load-bearing. BookStack mandates a client secret (`$required=['clientId','clientSecret','issuer']`) so no PKCE public-client escape. **Resolution options:** (1) `AUTH_METHOD=standard` in dev — unblock now, zero prod exposure, but dev never exercises real OIDC; (2) serve *dev* aithne over HTTPS via mkcert — correct root-cause fix, but aithne's issuer = single `APP_ORIGIN` (`oidc.go`), so it changes the issuer for EVERY dev OIDC consumer + needs CA trust in each consumer container ⇒ an **aithne dev-infra initiative**, not a worlds ticket. Recommended: Opt 1 now, Opt 2 as a separate strategic initiative (its payoff is estate-wide dev OIDC fidelity — the ES256/client-auth/email class of bug caught late in prod).

**lucas42's revive attempt + FINAL verdict (2026-07-09, with lucos-security):** lucas42 proposed scoping the worlds→aithne linked credential and allowlisting that scope for dev→prod, to make Option 3 pass the guard via an *enforced* path (not the hand-minted-client bypass). **Rejected — Option 3 dead for good.** Two independent reasons: (1) **the creds-link scope is INERT for the OIDC auth-code login flow** — verified in aithne `handleAuthCodeGrant` (oidc.go:277–395): the OIDC client is authed by *secret hash* (aithne's own client store; secret delivered via `CLIENT_KEYS` from the link *value*), and the issued token's scopes come from the logging-in *principal* (`GetActiveScopes`), NOT the link scope. So scoping the link is "**scope theatre**" — passes the guard's letter without bounding the credential; *worse* than an honest bypass because it reads-as-safe to the next reviewer. The `:read` guard's premise (scope==capability) is FALSE for an OIDC client secret (value/scope decoupled — see [[reference_creds_scope_keyvalue_independent]]). (2) `allScopesReadOnly` (creds storage.go:383/scopes.go:46) is a **blanket check, no allowlist** — the "policy" is Go code; carving an exception edits the estate-wide guard, first exception being a full OIDC secret vs the central IdP (precedent erosion). **Durable aithne fact worth reusing:** a creds-link scope does NOT gate aithne's OIDC user-login flow (only machine/client-credentials grants would). Don't try to scope-gate an OIDC client credential via a creds link.

**FINAL resolution (2026-07-09): lucas42 rejected all 3 options — needs zero-config-on-first-`docker compose up` dev auth.** Chosen: **standard auth in dev**. Verified facts (drove it live): a fresh BookStack DB auto-seeds `admin@admin.com`/`password` (Admin, id=1) on first migration regardless of auth method; with `AUTH_METHOD=standard` that login works out-of-the-box (302→/). Minimal design: compose currently *hardcodes* `AUTH_METHOD=oidc` → change to `${AUTH_METHOD:-oidc}` (fail-secure: unset→oidc, so prod never falls open to the default admin; `:-default` idiom is CI-safe per `${VERSION:-latest}`), set `AUTH_METHOD=standard` in **dev** creds (prod needs no change), document dev login in README. Zero-manual-step because the dev `.env` fetch is already mandatory (APP_KEY/DB_PASSWORD). Tradeoff accepted: dev doesn't exercise real OIDC (verified in prod). No zero-config way to keep real OIDC in dev (BookStack https-issuer+TLS-verify wall). Alt considered: derive AUTH_METHOD from `ENVIRONMENT` via init script (no creds value) — cleaner but more image logic.

**"Does dev need auth at all?" — verified NO auth-less path (2026-07-09):** BookStack has no `none` auth method (`standard` is the floor); `AUTH_AUTO_INITIATE` (auto-login) only works with an external IdP (n/a in dev); public/anon access is a runtime DB setting (`app-public`), view-only, not zero-config, no authoring/admin. Authoring+admin **require a logged-in user** — proven: unauthenticated `books/create`,`create-book`,`settings`,`settings/users` all 302→`/login`. So "no auth" collapses to "frictionless default user" = the auto-seeded admin. A custom dev auto-login (auto-auth when ENVIRONMENT=development) was rejected as a security smell (must never leak to prod) + diverges from prod more.

**Cleanup / deleting an OIDC client-secret linked cred (2026-07-09, verified):** deleting `KEY_LUCOS_AITHNE` (worlds→aithne OIDC client secret) from dev is **inert on the aithne side** — creds `deleteLinkedCredential` just deletes the row + posts loganne events; it does NOT rotate/regenerate any other credential. aithne `reconcileOIDCClients` (main.go), for a manifest client with no CLIENT_KEYS secret, logs "no CLIENT_KEYS secret for declared client … — skipping" and **continues booting** (no failure, other clients untouched); reconcile upserts but **never prunes**, so an already-registered client can linger in aithne's SQLite with an old secret hash nobody holds (inert). Compose tidy for unused OIDC creds: default the interpolations — `OIDC_ISSUER=${AITHNE_ORIGIN:-}`, `OIDC_CLIENT_SECRET=${KEY_LUCOS_AITHNE:-}` — verified that bare `${VAR}` unset warns ("variable is not set") but `${VAR:-}` (empty default) is silent. Re-enabling dev OIDC later needs a linked-cred **re-link** for the secret (regenerates + redelivers via CLIENT_KEYS), not just an .env line.
