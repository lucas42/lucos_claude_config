description: A delegated click handler that calls preventDefault() before checking whether the click landed on a native form control (checkbox/radio) will silently break that control's own click AND keyboard activation — verify empirically with a headless browser, not just by reading the spec.

  type: feedback

When a delegated click listener sits on an ancestor of a native `<input type="checkbox">` (or radio) and calls `e.preventDefault()` unconditionally before checking `e.target`, it breaks the checkbox's own native toggle — not just for that click, but for **keyboard Space-to-toggle too**, since a keypress dispatches through the identical click/defaultPrevented mechanism.

**Why:** per the HTML activation-behavior spec, a checkbox's `checked` toggle is a "pre-click" step that gets *reverted* if `event.defaultPrevented` is true by the end of the full dispatch (capture + target + bubble) — this applies regardless of which listener in the chain called `preventDefault()`, or when within that listener's execution it was called. A common bug shape: the handler checks `if (e.target.closest('.checkbox-class')) return;` (assuming "the native toggle must have already happened") but the `preventDefault()` a few lines earlier already silently undid it. Net effect: the checkbox looks completely inert to both mouse and keyboard, while a "click elsewhere on the tile" fallback path (built to give a bigger click target) still works and can mask the bug during casual manual testing.

**How to apply:** any PR that adds a delegated-click handler wrapping a native form control (checkbox, radio, even a nested `<button>`) should be checked for this pattern — does `preventDefault()` fire unconditionally, or only after confirming the click target is *not* the control itself? If the ordering is wrong, this is a concrete, one-line-reorder fix, not a "maybe" — REQUEST_CHANGES per [[feedback-robustness-gaps-block]].

**Verify empirically, don't just cite the spec.** `npm install playwright && npx playwright install chromium` works in this environment (no `--with-deps`, no root needed — the browser binary installs standalone even though `apt` is unreachable here). Build a minimal standalone HTML page using the PR's actual CSS (so real layout/hit-testing applies, not a toy repro) and the PR's actual handler JS, then drive it with `page.mouse.click(x, y)` at the *exact* control's bounding-box center (not `page.click(selector)`, which can resolve to the wrong point if elements overlap) and `page.keyboard.press('Space')` after `page.focus(selector)`. Confirmed via this exact technique on lucos_photos PR #487 (2026-07-24): the PR's own Playwright QA had exercised "click elsewhere on tile" but not a direct click on the checkbox or keyboard Space, so the break went unnoticed in their manual pass.

The fix pattern: check `e.target.closest('.checkbox-class')` **first**, and `return` in that branch *without* calling `preventDefault()` at all — let the native toggle proceed and rely on the existing delegated `change` listener to pick it up. Verify a checkbox nested inside a wrapping `<a href>` doesn't also trigger unwanted navigation once `preventDefault()` is removed for that branch — in Chromium at least, a nested interactive control "wins" activation over the ancestor anchor, so no separate navigation-suppression workaround was needed (verify this per-case though, don't assume it generalizes to every browser).
