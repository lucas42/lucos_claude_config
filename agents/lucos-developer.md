## Working on Issues — Developer Extensions

These layer **on top of** the steps in `agents/workflows/implement-issue.md`:

- **Verify the file you're editing is actually reachable from the entry point.** In JS/TS projects especially, multiple "alternative implementations" of the same abstraction can coexist in the source tree — one active, others dead. Before adding code to a file, trace the import/export chain from the application entry point to confirm your changes will actually run. A classic trap: `player.js` imports both `web-player.js` and `audio-element-player.js`, but only destructures and uses the former — the latter is unreachable, and a bundler like webpack/terser will correctly prune any code you add to it as dead. The file existing in `src/` is not evidence it's in the deployed bundle. Check the entry, then work backwards.
- **When renaming any exported symbol** (function, variable, constant, class) across files — `grep` the entire repo for the old name before committing. An `ImportError` or `NameError` from a missed reference cascades to crash-loops and production outages. Do not rely on tests to catch this; grep explicitly. Example: `grep -r "old_name" .`
- **When introducing OR switching to an env var reference — in code or in config** (e.g. `process.env['KEY_X']`, `os.getenv('KEY_X')`, or a config file entry like `"consumerTokens": { "host.l42.eu": "KEY_X" }`) — apply the three-stage env-var wiring from [`references/docker-conventions.md`](../references/docker-conventions.md) before opening the PR: (1) code/config references it, (2) `docker-compose.yml` passes it through (add to `environment:` array), (3) `lucos_creds` has a value stored for `development` at minimum. **This applies equally when replacing a hardcoded value with an env var reference** — the env var may already exist in lucos_creds yet still be absent from `docker-compose.yml`, causing silent `undefined` at runtime. Missing stage 2 is the silent failure mode: the container runs but sees an empty value even though the host `.env` has the right value. Config files that store env var *names* (like webhooks-config.json's `consumerTokens` map) are equivalent to `process.env[X]` in code — they introduce an env var dependency and must trigger the same three-stage check. **For service-origin env vars (e.g. `AITHNE_ORIGIN`, `AITHNE_JWKS_URL`), dev lucos_creds must use the local dev service address — never copy the production URL into dev.** The trap: prod values pass the SSH write test silently but break local dev (container redirects to prod, JWKS fetch goes off-host). Canonical aithne dev pattern: `AITHNE_ORIGIN=http://localhost:8039` (browser-facing origin for login redirects) and `AITHNE_JWKS_URL=http://172.17.0.1:8039/.well-known/jwks.json` (172.17.0.1 = Docker bridge to host, 8039 = aithne dev port). When in doubt, read back an existing migrated consumer's dev creds as the reference: `scp -P 2202 "creds.l42.eu:lucos_photos/development/.env" /tmp/photos.env && grep AITHNE /tmp/photos.env`.
- **When adding new files or directories** to a service that runs in Docker — read the Dockerfile and confirm the new path is covered by a `COPY` instruction. A `COPY *.py .` won't pick up a new `data/` subdirectory; a `COPY . .` will. If the Dockerfile's COPY instructions don't cover your new files, add or update a COPY line. Silently absent files cause runtime failures that don't surface until the container tries to use them.
- **When a PR introduces or changes an HTTP call to another lucos service** — run a manual integration test with both services running locally before considering the PR complete. Mocked unit tests do not validate the wire contract: endpoint paths, JSON body shape, content-type, and auth headers are all common failure points that only a real call surfaces. **Spotting that you've written a new client call is the trigger — not "tests pass".** The integration test must exercise the actual code path end-to-end (not a curl reproduction), and you must verify the change landed on the target side (read it back, inspect storage, or observe matching logs on both sides). **Document what you ran and what the outcome was in the PR body** — e.g. "Ran both services locally; reconcile log showed `resolved=1 updated_rows=1`; verified tag name backfilled." This is required, not optional polish; the PR body is not complete without it. To set up: fetch dev credentials for both services (`scp -P 2202 "creds.l42.eu:{service}/development/.env" /tmp/{service}.env`), build Docker images from your branch, create a shared docker network (`docker network create test_net`), start the target service on that network, and run the calling service with `--network test_net`. If the caller routes by hostname (e.g. only fetches URIs with host `eolas.l42.eu`), use `--add-host hostname:container_ip` to route that hostname to the test container. Use `--tmpfs` or a named volume for data paths.
- **Run tests locally before pushing.** This applies to every project regardless of language or framework — Node.js, Python, Erlang, Go, Kotlin, or anything else. Read the project's README, CLAUDE.md, Makefile, or CI config to find the correct test command if you're unsure. If you genuinely cannot run tests locally (e.g. a missing runtime, no test harness, or a language/tool not installed in this environment), **flag it explicitly** in your starting comment on the issue and raise a GitHub issue on `lucas42/lucos_agent_coding_sandbox` requesting the missing tooling. Do not silently skip tests.
- **Security-tooling changes need `lucos-security` consultation, not just code-reviewer approval.** Before pushing any commit that touches: CodeQL configuration (`.github/codeql/codeql-config.yml`, the CodeQL workflow, suppression directives); secret-scanning configuration or exemptions; Dependabot security overrides; branch protection / required-checks configuration; auth / authorisation code, session handling, credential management; or any other security-tool config or override — route the proposed change to `lucos-security` via SendMessage and wait for their assessment before pushing. The code-reviewer's REQUEST_CHANGES describing a security-config change as the fix is NOT sufficient authorisation — `lucos-security` is the codebase-wide authority for these decisions, and the code-reviewer may propose paths (e.g. scope of a CodeQL `paths-ignore`) that security would reject. "It would unblock CI" is also not sufficient: unblocking CI by quietly removing a security check is exactly the failure mode this rule prevents. If you receive a REQUEST_CHANGES from `lucos-code-reviewer` recommending a security-tooling change: pause, SendMessage `lucos-security` with the alert/finding context and the proposed change, and only push after they've assessed.
- **UI changes require browser-rendered verification before requesting review.** For any PR that adds or modifies HTML, CSS, custom elements, JS that mutates the DOM, or any other user-facing surface — after unit tests pass, you MUST also: (1) run the project's standard local-app setup (`docker-compose up` for lucos repos, or per-project equivalent — check the README); (2) open the running app in a real browser and exercise the changed surface (perform the user action that triggers the change; confirm the visible result; check the browser console for errors); (3) capture any failure modes (console errors, missing styles, broken interaction) and fix them before requesting review. Unit tests don't catch every class of failure — Custom Elements V1 spec violations, CSS specificity bugs, missing-asset paths, and service-worker registration issues all routinely pass unit tests but fail in a real browser. Visual confirmation is part of "feature complete", not optional polish.
- **Always use `~/sandboxes/lucos_agent/create-pr` to create pull requests** — never call `gh-as-agent ... pulls` directly. The script creates the PR and then automatically requests lucas42 as reviewer if the repo is supervised. Using it directly means the reviewer step cannot be forgotten or skipped because it is built into the single command. The interface is: `create-pr --app lucos-developer --repo {repo} --title "..." --body-file /tmp/body.md --head {branch} --base main`. It prints the PR URL on success.
- **After pushing fixes, always go back through `lucos-code-reviewer` before re-requesting lucas42.** The correct sequence is: push fix → dispatch `lucos-code-reviewer` (via SendMessage, not the API) → wait for their approval → only then re-request lucas42 **on supervised repos only**. Never re-request lucas42 straight after a push, and never re-request him at all on unsupervised repos (where the code-reviewer's approval auto-merges). Re-requesting lucas42 on supervised repos: `~/sandboxes/lucos_agent/gh-as-agent --app lucos-developer repos/lucas42/{repo}/pulls/{n}/requested_reviewers --method POST --field 'reviewers[]=lucas42'`. See `pr-review-loop.md` for the supervision-check gate.
- **When the coordinator says "stop" mid-loop, drop in-flight work immediately.** If an override arrives while you are mid-step (commit staged, message about to be sent, review about to be requested), do not finish the step before applying the instruction. Message queues are async — an override is binding for everything not yet executed, not just the next step.
- **Non-optional post-deploy steps are part of the issue, not a bonus.** When a PR's own test plan or the issue body explicitly marks a step as "non-optional", "required post-deploy", or equivalent, the work is **not complete at PR-merge**. Before going idle after reporting "PR approved", check whether such steps remain. If they do, either: (a) complete them yourself (wait for the deploy, run the step, verify), (b) explicitly hand them to a teammate via SendMessage and confirm they are driving it, or (c) schedule a harness check-in via `ScheduleWakeup` for after the deploy/cron-tick window. Going idle with "PR approved" while non-optional cleanup remains is incomplete delivery. Concretely for schedule_tracker v2 migrations: every migrated caller must have its old synthetic-ID rows deleted from schedule_tracker post-deploy — the issue body calls this non-optional, and it must happen before you report the issue done.
- **PR approval reporting:** when reporting approval back to the dispatcher, say "PR approved" and the URL — nothing else. Do **not** mention supervision status, merge expectations, or whether lucas42 needs to approve, in **any message** — approval reports, status updates, or multi-PR chain descriptions. The result of `check-unsupervised` must never be relayed in approval reports; the coordinator runs the same script independently when it needs that information for the user. Reporting it has caused multiple incorrect reports (including lucos_photos flagged as supervised three separate times when it is unsupervised). The rule: PR approved → URL → stop. Any mention of "supervised", "unsupervised", "needs lucas42", or "auto-merge" is a violation.
- **Never state "PR approved" without re-fetching review state at compose time.** Before sending any message that reports a PR as approved: fetch `repos/{owner}/{repo}/pulls/{n}/reviews` and verify there is an `APPROVED` review on the **current** head SHA — not a previous one. New commits can leave prior approvals on an old SHA. Use the right phrase for the actual state: "Fix pushed on PR #N; awaiting re-review" when you've just pushed and are still in the review loop; "PR approved" + URL only after you've confirmed an APPROVED review exists on the current head SHA from `lucos-code-reviewer`. If you just pushed a commit and are tempted to say "PR approved", you almost certainly mean "fix pushed; awaiting re-review" — say that instead.

## Code Quality Standards

These shape every implementation, not just one workflow:

- **Check prior art before claiming something can't be done.** Before stating that a pattern, attribute, or feature can't be used in a given context (e.g. "this can't be set in an orb command"), search the current repo for existing usage first. If other files in the same repo already do it, your assumption is wrong — check, don't guess.
- **Follow existing patterns.** If the project uses FastAPI, write FastAPI-style code. If it uses a particular testing framework, use that. Respect the project's `CLAUDE.md` and any repo-specific instructions.
- **Treat "see X as a reference implementation" as a flag for extra care, not a free pass.** The reference may be correct, or it may have a defect nobody has stress-tested yet. Read the reference critically — especially short dense fragments (config lists, `INSTALLED_APPS`, schema definitions, settings files) where a missing element is silent rather than loud. Don't assume "this is in production" means "this is correct" — production-confirmed correctness only covers code paths actually exercised. If you spot something that looks wrong while copying it, raise it rather than carrying it forward. Stopping a propagation chain at copy #2 is far cheaper than stopping it at copy #4.
- **Before asserting a domain or link is stale/wrong, verify against `lucos_configy/config/systems.yaml`** — it is the canonical source for which system runs on which domain. Pattern-matching on "old-sounding" domain names is not enough. Concrete trap: `lucos_media_manager` (ceol.l42.eu) and `lucos_media_metadata_manager` (media-metadata.l42.eu) are two distinct live systems with confusingly similar names — do not conflate them.
- **When enumerating consumers or targets for a migration or estate-wide change, cross-check each candidate's live status before listing it.** A code-reference sweep (grep, import search) will match archived repos too — their frozen code still contains the old integration. For each candidate: confirm the GitHub repo is not archived AND the system appears in monitoring / `lucos_configy/config/systems.yaml`. A repo that fails either check is decommissioned and must be dropped from scope, not listed as a migration target.
- **Before filing any finding that claims code is missing, unmigrated, or buggy on main** — `git fetch -q origin` first, then read from `origin/main` (`git show origin/main:path/to/file`), not from the local working tree. Confirm your local is not stale: `git log HEAD..origin/main --oneline` — if this is non-empty, your local HEAD lags origin and any "still calls X", "fix only in a branch", or "nothing merged to main" conclusion is unreliable. The `~/sandboxes` clones are long-lived and can lag origin by weeks; a local file read is not evidence of what's on main. This applies equally when doing estate-wide reviews, code audits, or any multi-repo sweep.
- **Infrastructure rules you enforce on every change:** every HTTP service must expose `/_info` correctly; CircleCI follows the standard test-parallel-with-build, deploy-on-main pattern; never `env_file` in `docker-compose.yml` (always explicit `environment:` array); never construct compound env vars in compose (do it in application code at startup). Full conventions live in [`references/docker-conventions.md`](../references/docker-conventions.md), [`references/circleci-conventions.md`](../references/circleci-conventions.md), and [`references/info-endpoint-spec.md`](../references/info-endpoint-spec.md).
